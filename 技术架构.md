## 1. 项目概述

**Life Reflex Arc** 是一套面向突发心脏骤停场景的分布式协同急救系统。系统通过移动 App 自动将附近目击者分配为不同角色，实现 CPR 施救、AED 取送、救护车引导三线并行，最大程度缩短从发现患者到完成急救交接的时间窗口。

### 核心角色

| 角色 | 英文 | 职责 |
|------|------|------|
| 主施救者 | PRIME | 原地对患者实施 CPR，使用 App 节拍器保持按压频率 |
| 取机员 | RUNNER | 奔赴最近 AED 取回设备并交付 |
| 引导员 | GUIDE | 在路口等待救护车并引导进场，完成交接存档 |

### 核心事件流

```
患者倒下
  └─> SOS 倒计时启动（10 s 未取消则自动触发）
        └─> 系统向附近用户推送急救邀请
              └─> 用户加入并被分配角色（PRIME / RUNNER / GUIDE）
                    ├─> PRIME：CPR 开始 → AED 到达后使用 AED
                    ├─> RUNNER：取 AED → 送达现场（AED_PICKED → AED_DELIVERED）
                    └─> GUIDE：等待救护车 → 引导到场 → 交接存档（HANDOVER）
```

### Demo 局限性

| 问题 | 影响 |
|------|------|
| 数据存储在内存字典中 | 服务重启所有事件数据丢失 |
| 单进程、无集群 | 无法水平扩展，单点故障 |
| WebSocket 连接存储在进程内 | 多实例时消息无法广播到跨进程客户端 |
| 无用户认证 | 任意人可操控任意事件 |
| 无推送通知 | App 需主动轮询，无法后台唤醒附近用户 |
| 无地理位置服务 | 无法真正定位最近 AED 或附近响应者 |
| 无 AED 数据库 | RUNNER 无法获得实际 AED 坐标和导航路线 |
| 无日志/监控 | 生产问题无法追溯 |

| 组件 | 作用 |

### 4.3 业务服务层（微服务）

| 服务 | 接口前缀 | 主要职责 |
|------|----------|----------|
| 事件服务 (Incident Service) | `/api/v1/incidents` | 事件 CRUD、状态机、角色分配、SOS 倒计时 |
| 用户服务 (User Service) | `/api/v1/users` `/api/v1/auth` | 注册/登录、JWT 颁发、用户档案 |
| 地理位置服务 (Geo Service) | `/api/v1/geo` | 最近 AED 查询、响应者地理围栏、导航路线 |
| 通知服务 (Notification Service) | `/api/v1/notify` | FCM 推送、短信、站内消息队列 |
| 存档服务 (Archive Service) | `/api/v1/archive` | 事件完整记录、HANDOVER 报告生成 |

### 4.4 数据层

| 存储 | 用途 |
|------|------|
| PostgreSQL | 持久化事件、用户、AED 位置（PostGIS 扩展支持地理查询） |
| Redis | WebSocket 连接注册表、SOS 倒计时分布式锁、会话缓存 |
| MinIO / OSS | HANDOVER 存档文件、事件回放日志 |

## 5. 核心模块详解

### 5.1 事件状态机

```
                ┌────────────┐
          reset │            │ create
                ▼            │
           ┌─────────┐       │
           │ CREATED │───────┘
           └────┬────┘
                │ join / sos auto-trigger
                ▼
          ┌──────────┐
          │DISPATCHED│
          └────┬─────┘
               │ CPR_STARTED
               ▼
           ┌───────┐
           │  CPR  │
           └───┬───┘
               │ AED_PICKED
               ▼
          ┌──────────┐
          │ AED_PICKED│
          └─────┬─────┘
                │ AED_DELIVERED
                ▼
         ┌─────────────┐
         │ AED_DELIVERED│
         └──────┬───────┘
                │ AMBULANCE_ARRIVED
                ▼
           ┌─────────┐
           │ HANDOVER │
           └──────────┘
```

每个状态转换均需：
1. 校验发起者的角色权限
2. 写入 PostgreSQL（事务保证原子性）
3. 通过 Redis Pub/Sub 广播给 WebSocket Gateway

### 5.2 SOS 倒计时（分布式实现）

Demo 中使用 `asyncio.sleep` 实现倒计时，多实例时会重复触发。生产方案：

```
用户按下 SOS
  → 事件服务向 Redis 写入 EXPIRE Key：sos:{incidentId} = startTs，TTL=10s
  → Redis Keyspace Notification 在 Key 过期时触发 Subscriber
  → Subscriber 调用事件服务内部接口完成状态转换
  → 广播状态更新
```

### 5.3 角色分配算法（生产版）

Demo 中使用 `random.choice`，生产版需考虑：

1. **距离优先**：优先分配物理距离最近且健康的用户
2. **能力过滤**：PRIME 优先选经过 CPR 培训认证的用户
3. **负载均衡**：排除近期已参与多次任务的用户（防止疲劳）
4. **分配超时重推**：若被邀请用户 30 s 内未响应，自动邀请下一位候选人

```python
# 伪代码
candidates = geo_service.find_nearby_users(
    lat=incident.lat, lng=incident.lng, radius_m=500
)
ranked = role_ranker.rank(candidates, role=role, incident=incident)
for user in ranked:
    ok = notify_service.invite(user, incident, role, timeout_sec=30)
    if ok:
        assign(incident, user, role)
        break
```

## 6. 数据库设计

### 6.1 核心表结构（PostgreSQL）

```sql
-- 用户表
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    phone       VARCHAR(20) UNIQUE NOT NULL,
    name        VARCHAR(100),
    cpr_certified BOOLEAN DEFAULT FALSE,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- 事件表
CREATE TABLE incidents (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    phase       VARCHAR(30) NOT NULL DEFAULT 'CREATED',
    lat         DOUBLE PRECISION,
    lng         DOUBLE PRECISION,
    location    GEOGRAPHY(POINT, 4326),  -- PostGIS
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW(),
    closed_at   TIMESTAMPTZ
);

-- 角色分配表
CREATE TABLE incident_roles (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    incident_id UUID REFERENCES incidents(id),
    user_id     UUID REFERENCES users(id),
    role        VARCHAR(10) NOT NULL,  -- PRIME / RUNNER / GUIDE
    status      VARCHAR(30) NOT NULL,
    assigned_at TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE (incident_id, role)  -- 同一事件每个角色同时只能由一人持有；
                                -- 若用户拒绝或 30s 超时未响应，系统将该记录
                                -- 的 status 更新为 REJECTED/TIMEOUT 并释放角色，
                                -- 再向下一位候选人发送邀请（见第 5.3 节角色分配算法）
);

-- 事件日志表
CREATE TABLE incident_logs (
    id          BIGSERIAL PRIMARY KEY,
    incident_id UUID REFERENCES incidents(id),
    user_id     UUID,
    action      VARCHAR(50),
    detail      JSONB,
    ts          TIMESTAMPTZ DEFAULT NOW()
);

-- AED 位置表（PostGIS）
CREATE TABLE aed_locations (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        VARCHAR(200),
    location    GEOGRAPHY(POINT, 4326) NOT NULL,
    address     TEXT,
    floor       VARCHAR(20),
    is_active   BOOLEAN DEFAULT TRUE,
    last_checked TIMESTAMPTZ
);
CREATE INDEX ON aed_locations USING GIST(location);

-- 事件索引
CREATE INDEX ON incidents USING GIST(location);
CREATE INDEX ON incidents(phase);
CREATE INDEX ON incidents(created_at DESC);
```

### 6.2 Redis 数据结构

| Key 模式 | 类型 | 用途 | TTL |
|----------|------|------|-----|
| `session:{token}` | Hash | JWT 会话缓存 | 7d |
| `ws:incident:{id}:subs` | Set | 订阅该事件的 WS 连接 ID | - |
| `sos:{incident_id}` | String | SOS 倒计时触发器 | 10s |
| `incident:{id}:state` | JSON | 事件状态热缓存 | 5min |
| `geo:users` | Sorted Set (GEO) | 在线用户地理位置 | - |

## 7. API 设计规范

### 7.1 RESTful API

所有接口遵循 REST 规范，统一前缀 `/api/v1`。

#### 认证接口

```
POST   /api/v1/auth/register       注册（手机号 + 验证码）
POST   /api/v1/auth/login          登录，返回 JWT
POST   /api/v1/auth/refresh        刷新 Token
POST   /api/v1/auth/logout         注销
```

#### 事件接口

```
POST   /api/v1/incidents                    创建事件
GET    /api/v1/incidents/current            获取当前事件
POST   /api/v1/incidents/current/reset      重置当前事件
GET    /api/v1/incidents/{id}               获取事件详情
POST   /api/v1/incidents/{id}/sos_start     触发 SOS 倒计时
POST   /api/v1/incidents/{id}/sos_cancel    取消 SOS 倒计时
POST   /api/v1/incidents/{id}/join          加入事件（指定角色）
POST   /api/v1/incidents/current/join_auto  加入当前事件（自动分配角色）
POST   /api/v1/incidents/{id}/actions       上报行动（CPR_STARTED 等）
POST   /api/v1/incidents/{id}/trigger       手动触发事件
GET    /api/v1/incidents/{id}/logs          获取事件日志
```

#### 地理位置接口

```
POST   /api/v1/geo/location                 上报当前位置

GET    /api/v1/geo/route?from=&to=          获取导航路线
```

## 8. 实时通信架构

Demo 中，`server.py` 将所有 WebSocket 连接以 `Dict[str, List[WebSocket]]` 的形式存储在单进程内存中（`ws_connections` 变量）。生产方案需解决多实例部署时跨进程广播的问题。

### 8.1 架构方案

```
Android App / Web 看板
        │  WebSocket 长连接
        ▼
┌────────────────────────────────┐
│    WebSocket Gateway Pod 组    │
│  (多副本，每个维护本地连接池)    │
└──────────────┬─────────────────┘
               │ 订阅
               ▼
        ┌─────────────┐
        │  Redis      │
        │  Pub/Sub    │◄─── 事件服务发布状态变更
        └─────────────┘
```

### 8.2 消息格式（WebSocket）

```json
// 服务器 → 客户端：状态推送
{
  "type": "STATE",
  "payload": {
    "incidentId": "uuid",
    "phase": "CPR",
    "sos": { "status": "ALERTING", "startTs": 1700000000000, "durationSec": 10 },
    "roles": {
      "PRIME":  { "status": "CPR_STARTED", "userId": "u1" },
      "RUNNER": { "status": "AED_PICKED",  "userId": "u2" },
      "GUIDE":  { "status": "JOINED",      "userId": "u3" }
    },
    "logs": [...]
  }
}

// 服务器 → 客户端：错误通知
{ "type": "ERROR", "payload": "Incident not found" }

// 客户端 → 服务器：心跳
{ "type": "PING" }

// 服务器 → 客户端：心跳回包
{ "type": "PONG" }
```

### 8.3 连接生命周期

```
客户端连接 → 身份验证（JWT via query param）
         → 订阅事件 incidentId
         → 收到当前完整状态快照
         → 持续接收增量 STATE 更新
         → 每 30s 发送 PING，超时未回应则断开重连
```

## 9. 安全与认证

### 9.1 认证方案

采用 **手机号 + 短信验证码** 无密码认证，颁发 JWT（Access Token 30min + Refresh Token 7d）。

```
手机号注册/登录
  → 发送短信验证码（阿里云 SMS / Twilio）
  → 校验验证码
  → 颁发 JWT (HS256，含 userId、role、exp)
  → Refresh Token 存入 Redis，用于无感续期
```

### 9.2 权限控制

| 操作 | 所需权限 |
|------|----------|
| 创建/重置事件 | 已登录用户 |
| 加入事件 | 已登录用户 |
| 上报 CPR_STARTED | 当前事件 PRIME 角色 |
| 上报 AED_PICKED / AED_DELIVERED | 当前事件 RUNNER 角色 |
| 上报 AMBULANCE_ARRIVED | 当前事件 GUIDE 角色 |
| 查看调度看板 | DISPATCHER 角色（运营人员） |

### 9.3 安全加固

- HTTPS 全站强制（TLS 1.3）
- API Gateway 层 JWT 验证，拒绝无效请求下发到服务层
- 限流：同一 IP 10 req/s；同一用户 5 req/s（Redis 令牌桶）
- WebSocket 连接建立时校验 JWT，过期则强制断开
- 敏感操作（SOS 触发、角色变更）写入审计日志
- SQL 注入防护：ORM 参数化查询（SQLAlchemy / Tortoise ORM）
- 用户手机号存储时脱敏（仅展示前3后4位）

## 10. 推送通知

Demo 无推送能力，用户需主动打开 App 才能看到邀请。生产方案：

### 10.1 推送渠道

| 场景 | 渠道 | 优先级 |
|------|------|--------|
| 附近有事件，邀请加入 | FCM / APNs 高优先级推送 | 紧急 |
| 角色分配成功 | FCM 数据消息 | 高 |
| 事件状态变更 | FCM 数据消息 | 正常 |
| HANDOVER 存档就绪 | FCM 通知消息 | 普通 |

### 10.2 推送流程

```
事件服务 → 通知服务（gRPC）
        → 查询附近在线用户（Redis GEO）
        → 批量调用 FCM API（Android）
        → 设备收到推送 → 打开 App → 自动加入事件
```

### 10.3 地理围栏触发

```
用户 App 每 60s 上报位置到 Redis GEO：
  GEOADD geo:users <lng> <lat> <userId>

事件创建时：
  通知服务查询半径 500m 内在线用户
  GEORADIUS geo:users <lng> <lat> 500 m ASC COUNT 20
  → 向候选用户发送 FCM 推送邀请
```

## 11. AED 位置服务

### 11.1 数据来源

| 来源 | 说明 |
|------|------|
| 政府开放数据 | 各城市 AED 公开数据集（CSV/API） |
| 医疗机构合作 | 医院、购物中心、交通枢纽等提供坐标 |
| 用户众包上报 | App 内"上报 AED"功能，经人工审核入库 |
| 第三方 AED 数据库 | 如 PulsePoint（国际）、各地 120 系统 |

### 11.2 AED 查询接口（PostGIS）

```sql
-- 查询半径 1km 内最近的 5 台可用 AED
SELECT id, name, address, floor,
       ST_Distance(location, ST_MakePoint(:lng, :lat)::geography) AS dist_m
FROM aed_locations
WHERE is_active = TRUE
  AND ST_DWithin(location, ST_MakePoint(:lng, :lat)::geography, 1000)
ORDER BY dist_m ASC
LIMIT 5;
```

### 11.3 导航路线

RUNNER 获得 AED 坐标后，App 调用地图 SDK 规划步行路线：
- 国内：高德地图 SDK（步行导航）
- 国际：Google Maps Directions API

```
GET /api/v1/geo/route?fromLat=&fromLng=&toLat=&toLng=&mode=walking
→ 返回 encoded polyline + 步行距离 + 预计时间
```

## 14. 扩展性与高可用设计

### 14.1 水平扩展策略

| 服务 | 扩展方式 | 状态共享 |
|------|----------|----------|
| 事件服务 | 多副本（无状态） | 状态存 PostgreSQL + Redis |
| WebSocket 网关 | 多副本 | 连接注册表存 Redis，消息通过 Pub/Sub 广播 |
| 通知服务 | 多副本（无状态） | 消息队列（RabbitMQ / Kafka）保证不重复推送 |
| 地理位置服务 | 多副本（无状态） | 位置数据存 Redis GEO |

### 14.2 数据库高可用

```
PostgreSQL 主从复制
  主节点：写操作（事件状态变更、日志写入）
  从节点：读操作（日志查询、看板数据）
  PgBouncer：连接池，降低连接数压力

Redis Cluster（3 主 3 从）
  自动分片、故障转移
  Sentinel 监控主节点可用性
```

### 14.3 容灾策略

- 多可用区部署（AZ-A / AZ-B / AZ-C）
- PostgreSQL 开启跨 AZ 同步复制
- 定时全量备份（每日）+ WAL 增量备份（实时）
- RTO（恢复时间目标）< 5 分钟，RPO（恢复点目标）< 1 分钟
